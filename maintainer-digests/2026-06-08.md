## Step 3 — Gate Decision

**Result: GATE FAILED. Build nothing this week.**

The single highest-Final candidate (#1, 40.0) clears the numeric threshold (≥ 8) and
carries no multi-run flag, but **fails the third gate criterion** — "a self-contained
change you can complete cleanly in one run" *with genuine value*.

Verification (read-only) of the candidate before gating:
- `BaseAgent` and its subclasses (`DrafterAgent`, `CriticAgent`, …) are imported in
  exactly one place: re-exported by `src/llm_council/registry/__init__.py`. They are
  **never instantiated or called** anywhere in `src/` or `tests/`.
- The `create_agent()` factory in the same module is likewise **never called**.
- The production 3-phase flow runs in `engine/orchestrator.py`, directly against
  provider adapters — it does not use the `registry` agent abstraction at all.
- Per `docs/architecture/capability-augmented-council.md` (P1–P1.5, in progress),
  these `run()` stubs are scaffolding for an unfinished larger feature, not broken
  production code.

Implication: wiring `run()` to provider dispatch **in isolation** would create a
second execution path nothing consumes — a divergent duplicate of the orchestrator
(speculative infrastructure / tech debt). Delivering real value requires *also*
refactoring the orchestrator to route through these agents, which is a multi-step
design effort, **not** cleanly completable in one run. The scout's RICE of 40.0
(low Effort × high Confidence) misreads "unused scaffolding" as "broken stub,"
inflating the score.

### One-paragraph implementation plan for the top item (for a future, scoped effort)

Treat this as a deliberate, multi-step feature — not a one-run patch. (1) Decide the
*consumer*: either make `engine/orchestrator.py` construct council members via
`registry.create_agent(role, model, …)` and drive its phases through `BaseAgent.run()`,
or formally mark the `registry` package as experimental and gate it behind a config
flag. (2) Implement `BaseAgent.run()` as a real model+tool loop: call the selected
`ProviderAdapter` with `self.system_prompt` + `AgentContext`, run up to
`max_iterations` tool-call turns against `ToolRegistry`, apply `_detect_pass`, and
populate `AgentResult.usage`/`tool_calls`/`metadata`. (3) Have each subclass override
only role-specific assembly, reusing the base loop. (4) Land unit tests with a mocked
provider per role (this also clears candidate #2's coverage gap for `base_agent.py`).
(5) Only then wire the orchestrator over to it behind a flag, with a regression test
proving output parity with the current direct-adapter path before any default switch.
